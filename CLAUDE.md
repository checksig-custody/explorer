# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Fork of [ping.pub explorer](https://github.com/ping-pub/explorer) — a "light" multi-chain explorer for Cosmos-SDK blockchains. The app talks **directly** to each chain's public LCD (REST) / RPC endpoints from the browser; there is no backend, no DB, and no caching layer. Everything you see in the UI is a live request against the configured node.

## Commands

Package manager is **yarn** (pinned via `packageManager`). Node engine checks are typically bypassed (see `installation.md`).

```sh
yarn --ignore-engines        # install
yarn dev                     # vite dev server
yarn build                   # type-check + vite build (run-p)
yarn build-only              # skip type-check
yarn type-check              # vue-tsc --noEmit
yarn format                  # prettier --write .
yarn preview                 # preview the production build
```

There are no tests in this repo.

## High-level architecture

### Chain configuration is the entry point
- Chain configs live in `chains/mainnet/*.json` and `chains/testnet/*.json`. They are loaded at build time via `import.meta.glob` in `src/stores/useDashboard.ts`.
- **Which folder is loaded depends on the browser hostname**: if `window.location.hostname` contains the substring `testnet`, only `chains/testnet/*` is loaded; otherwise only `chains/mainnet/*`. See `loadingFromLocal()`. The `chains/README.md` documents this and the JSON schema.
- `LocalChainConfig` (file shape) is normalized to `ChainConfig` (in-app shape) by `convertFromLocal()` in `useDashboard.ts`. Types live in `src/types/chaindata.ts`.
- Note: `chains/Kiichain` is a stray top-level JSON file (not under `mainnet/`/`testnet/`) and therefore is **not** loaded by the glob. If you want it active, move it into the right network folder.

### Routing is file-based with a `[chain]` segment
- Routes are generated by `vite-plugin-pages` from `src/pages/` and `src/modules/` (see `vite.config.ts`).
- The dynamic chain segment is the directory `src/modules/[chain]/` — every subfolder under it (`account`, `block`, `gov`, `staking`, `ibc`, `cosmwasm`, `tx`, `uptime`, ...) becomes `/<chain>/<module>` routes. `[address].vue` / `[validator].vue` give nested dynamic params.
- `src/router/index.ts` adds a global `beforeEach` guard that calls `useBlockchain().setCurrent(chain)` whenever the `:chain` param changes — switching chains is essentially "navigate to a different URL prefix."
- Layouts come from `src/layouts/` via `vite-plugin-vue-layouts`.

### State: Pinia stores in `src/stores/`
- `useDashboard` — loads/normalizes chain configs, tracks favorites, fetches CoinGecko prices.
- `useBlockchain` — the "currently selected chain" store. Holds the active `CosmosRestClient` endpoint, applies `themeColor` to CSS variables, builds the sidebar nav by filtering router routes against `current.features`. Most module pages read from here.
- Domain stores (`useBankStore`, `useStakingStore`, `useGovStore`, `useMintStore`, `useDistributionStore`, `useParamsStore`, `useBaseStore`, `useWalletStore`, `useTxDialog`, `useFormatter`, `useCoinGecko`) each wrap a slice of REST endpoints. They depend on `useBlockchain` for the active client.
- `src/main.ts` sets up a global `setInterval` that calls `useBaseStore().fetchLatest()` every `VITE_REFRESH_INTERVAL` ms (default 6s) with a max-in-flight cap of 5.

### REST client + per-chain customization (`src/libs/`)
This is the trickiest part of the codebase.

- `src/libs/api/index.ts` defines `DEFAULT: RequestRegistry` — a map from logical endpoint name (e.g. `gov_proposals`) to `{ url, adapter }`. URLs use `{placeholder}` interpolation.
- `src/libs/api/registry.ts` defines the typed registry interface, two profile lookups (`NAME_REGISTRY` by chain name, `VERSION_REGISTRY` by Cosmos SDK semver) and helpers `registryChainProfile` / `registryVersionProfile` / `findApiProfileByChain` / `findApiProfileBySDKVersion`.
- `src/libs/api/customization/*.ts` files each export `{ store: 'name' | 'version', name, requests: Partial<RequestRegistry> }`. **They are auto-discovered** by `registeCustomRequest()` in `src/libs/client.ts` via `import.meta.glob('./api/customization/*.ts', { eager: true })`. There is **no central import list** — adding a file to that folder is enough.
- `CosmosRestClient.newStrategy(endpoint, chain)` picks the profile: first by `chain.chainName`, then by SDK version (latest profile `<=` the chain's `cosmosSdk` per semver). The selected version is also cached per chain in `localStorage` under `sdk_version_<chainName>`.
- `BaseRestClient.request()` does the URL interpolation and dispatches to `fetchData()` / a per-request `adapter` — adapters are where you reshape weirdly-versioned responses (e.g. gov v1 vs v1beta1 tally fields).

**When a chain's API diverges from the cosmos-sdk defaults, add a customization file rather than special-casing call sites.** See `osmosis.ts`, `xion.ts`, `nolus.ts`, `evmos.ts`, `atomone.ts`, `v0.46.0.ts`, `v0.50.0.ts` for the two patterns (`store: 'name'` vs `store: 'version'`).

### Vue tooling specifics
- `unplugin-auto-import` auto-imports from `vue`, `vue-router`, `@vueuse/core`, `@vueuse/math`, `vue-i18n`, `pinia` — types regenerated into `auto-imports.d.ts`. Don't be surprised by missing imports in `.vue` files; they're injected at build time.
- `unplugin-vue-define-options` enables `<script>` block `defineOptions()`.
- Three custom elements are whitelisted in the Vue compiler: `ping-connect-wallet`, `ping-token-convert`, `ping-tx-dialog` (web components from the ping ecosystem).
- TS path alias: `@/* → ./src/*`.

### i18n
- Locales: `src/plugins/i18n/locales/{de,en,es,id,ja,ko,zh}.json`. Sidebar menu items use keys like `module.<i18n-meta>` driven by each route's `meta.i18n` field — so when adding a new module page, set `definePage({ meta: { i18n: 'foo', order: 50 } })` (the menu won't show otherwise).

## CI / deploys
- `.github/workflows/testnet-deploy.yaml` deploys on pushes to the `testnet` branch via a self-hosted runner labeled `testnet`. It runs `yarn run vue-cli-service build` (note: stale — this project uses Vite, not vue-cli — the command likely fails; fix if you rely on this).
- `.github/workflows/mainnet-deploy.yaml.disabled` is intentionally disabled.
- `.github/workflows/config-check.yaml` runs `ping-pub/ping-dashboard-config-checker` on PRs that touch `chains/mainnet/**` or `chains/testnet/**`.

## Conventions worth knowing
- Prettier: single quotes, semicolons, 2-space indent, `printWidth: 80`, `trailingComma: 'es5'`. A husky `pre-commit` hook runs `pretty-quick --staged`.
- Snake_case is used for REST response shapes (matches Cosmos APIs); camelCase is used for store/getter names.
- Don't add caching of chain RPC responses — staying "light" (no caching, no pre-processing) is the explicit project philosophy stated in `README.md`.
